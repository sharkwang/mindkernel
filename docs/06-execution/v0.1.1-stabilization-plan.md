# MindKernel v0.1.1 稳定化执行计划（Stabilization Plan）

_创建时间：2026-02-25（Asia/Shanghai）_

> 背景：`v0.1.0-usable` 已完成并通过 14/14 总检。下一阶段目标从“可用”转向“可持续治理 + 可运维稳定性”。

---

## 0. 目标定义（DoD）

当以下条件全部满足，判定 v0.1.1 稳定化完成：

1. 周期治理可观测：形成可复用 weekly governance report（含趋势与异常）。
2. 调度并发更稳：lease 具备续约能力，长任务不误回收。
3. 遗忘执行闭环增强：temporal worker 覆盖 verify/revalidate 关键动作。
4. CI 门禁与本地总检一致：新增治理验证进入主 workflow（按时长分层）。
5. 检索演进有决策依据：完成向量检索补充评估并输出 go/no-go 结论。

---

## 1. 工作分解（R1~R6）

### R1. Weekly Governance Report（P2 主线）

- **输入**：`scheduler_jobs` / `audit_events` / validation reports
- **动作**：
  1. 定义周报指标字段（成功率、重试率、dead-letter、回滚率、升级率、学习收益 proxy）；
  2. 新增周报生成脚本（JSON + Markdown）；
  3. 固化报告目录与命名规则。
- **输出**：`tools/validation/generate_weekly_governance_report_v0_1.py`（或等价实现）
- **验收**：连续两次生成成功，字段稳定可比对。

### R2. Scheduler Lease Renew（长任务保护）

- **输入**：`tools/scheduler/scheduler_v0_1.py`
- **动作**：
  1. 增加 lease renew 接口（CLI + 函数）；
  2. worker 可按心跳续租；
  3. 增加过期/续租冲突审计事件。
- **输出**：renew 能力 + 回归测试
- **验收**：长任务场景不被误回收；冲突场景可审计。

### R3. Temporal Worker 动作扩展

- **输入**：`tools/scheduler/temporal_governance_worker_v0_1.py`
- **动作**：
  1. 覆盖 `verify/revalidate` 执行动作；
  2. 对齐 state-machines 合法迁移约束；
  3. 增加对应验证脚本与 fixture。
- **输出**：扩展后 temporal worker + validation
- **验收**：新增动作在回放与总检中稳定通过。

### R4. CI 门禁对齐（治理验证入主流水线）

- **输入**：`.github/workflows/critical-path-validation.yml`
- **动作**：
  1. 将 `validate_scheduler_multi_worker_lock_v0_1.py`、`validate_temporal_governance_worker_v0_1.py` 纳入 CI；
  2. `validate_scheduler_workspace_replay_v0_1.py` 作为可选/夜间门禁（避免 PR 时长过长）；
  3. 更新 release_check quick/full 与 CI 分层映射说明。
- **输出**：更新后的 workflow 与门禁文档
- **验收**：PR 时长可控、主门禁覆盖不回退。

### R5. 调度吞吐与延迟基线

- **输入**：scheduler sqlite + worker 运行结果
- **动作**：
  1. 制定最小基线（jobs/min、lag p95、retry rate）；
  2. 增加压测/回放脚本；
  3. 输出首版 benchmark 报告。
- **输出**：`reports/benchmark/*.md|json`
- **验收**：有可重复执行的基线报告。

### R6. 向量检索补充评估（P2）

- **输入**：现有 FTS recall 基线 + 代表性 query 集
- **动作**：
  1. 明确接入阈值（规模/延迟/质量收益）；
  2. 进行小样本 A/B（FTS vs FTS+vector）；
  3. 输出 go/no-go 决策文档。
- **输出**：评估报告 + 决策结论
- **验收**：给出明确落地或暂缓结论与理由。

## 3. 风险与策略

1. **工程时长风险**：R2/R3/R4 互相耦合，建议先做 R4 最小门禁，再迭代 R2/R3。
2. **CI 时长风险**：workspace replay 不宜强制每次 PR 执行，建议 nightly 或手动门禁。
3. **指标失真风险**：R5 基线需固定样本与环境，避免横向比较失真。
4. **技术债回流风险**：所有新增能力必须配套 test + validation + docs 同步，避免“只写脚本不入规范”。
